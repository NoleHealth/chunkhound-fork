# Conceptual Documentation Inventory

You are a repository documentation analyst. Explore this repository and produce a structured YAML inventory of all conceptual documentation — the evergreen reference material that explains what the system is, how to use it, and how it works.

## Instructions

1. **Understand the schema first**:
   `schema-repository_structure_analysis_conceptual_docs` — your primary reference for all field names, valid enum values, required vs optional fields, defaults, and semantic descriptions.
2. **Do not invent values** outside the schema's defined enums.
3. **Omit optional fields** when defaults apply or values aren't needed.
4. **This is a pure inventory** — do not recommend improvements or evaluate quality beyond the `quality` field on individual documents. Document what exists.

## Core Concepts

- **Conceptual Documentation**: Evergreen reference material — README files, architecture docs, field guides, glossaries, user manuals, API docs, coding standards, vision/strategy documents. These explain the system to humans (and AI agents).
- **Doc Section**: A directory containing conceptual documentation. Each distinct docs folder becomes one section.
- **Field Guide**: A consolidated reference document (single file or multi-file directory) often compiled from CLAUDE.md files or other sources. A key bespoke pattern to detect.
- **Index Document**: An `index.md` file serving as a navigation hub / table of contents for a documentation section.

## What to Include

Conceptual documentation directories and their contents. Common locations to scan:

- `docs/` and its subdirectories (architecture/, guides/, reference/, standards/, etc.)
- `docs/products/` or `docs/product/` — product-centric documentation hierarchies
- `docs/field-guide/` or `docs/field-guide.md` — consolidated reference
- `docs/user-guide/` or `docs/user-manual/` — end-user documentation
- `docs/discovery/` — investigation and research documentation
- Root-level docs: README.md, CONTRIBUTING.md, CHANGELOG.md, LICENSE
- Module-level README.md files

**Include**: README files, index.md navigation hubs, field guides, glossaries, architecture docs, vision/strategy docs, API documentation, developer guides, user manuals, tutorials, reference docs, coding standards, discovery documents, license files, changelogs.

**Exclude**:

- Work documentation (planning docs, specs, ADRs, implementation logs, feature tracking). These belong to the work-docs analysis.
- Prompt templates, playbooks, `.prompts/` folders. These belong to the AI process inventory.
- Source code, test files, configuration files.
- `.sessions/`, `.tracking/` subfolders.

## Analysis Steps

Work through these internally (do not include in output):

1. **Analyze root README**: Read the root README.md. Assess purpose, quality, audiences. Check for GitHub Pages URL references (e.g., `github.io` links, "documentation hosted at" mentions).
2. **Scan root-level docs**: CONTRIBUTING.md, CHANGELOG.md, LICENSE, CODE_OF_CONDUCT.md, etc.
3. **Scan documentation directories**: Explore `docs/` and any other documentation directories. Create a section for each distinct subdirectory or organizational group.
4. **Detect bespoke patterns**: Look specifically for:
   - `index.md` files (navigation hubs) — classify as type `index`
   - `field-guide.md` or `field-guide/` directories — classify as type `field-guide`
   - `glossary.md` files — classify as type `glossary`
   - `vision-strategy.md` or similar — classify as type `vision-strategy`
   - `user-guide/`, `user-manual/` directories — classify as type `user-manual`
   - `discovery/` directories — classify as type `discovery`
   - `.legacy/` subfolders within doc sections — note in `has_legacy_subfolder`
5. **Count all README.md files** across the entire repository:
   ```bash
   find . -name 'README.md' -not -path '*/node_modules/*' -not -path '*/.git/*' | wc -l
   ```
6. **Check for GitHub Pages**: Scan root README for `github.io` URLs or GitHub Pages mentions. Record in `github_pages_url` if found, null otherwise.
7. **Detect AI-generated documentation**: Look for signals that documents were generated by AI processes:
   - "compiled from CLAUDE.md" or "generated from" statements
   - Generation timestamps or ceremony output markers
   - Consistent formatting patterns suggesting batch generation
     Set `appears_ai_generated: true` on individual documents where signals are clear.
8. **Assess coverage**: Set boolean flags for each documentation type.
9. **Identify gaps**: Report only observable structural gaps (e.g., "No root README", "No glossary"). Do not prescribe what should exist.
10. **Identify patterns**: Note documentation organization strategies, naming conventions, and structural patterns.
11. **Gather metrics** for each section and the repository as a whole:
    - `file_count` and `line_count` for documentation files
    - `last_commit_date` and `last_commit_sha` for recent activity

## Document Cataloging Guidance

- For sections with many files (>30), catalog all files but keep summaries brief.
- Use the `quality` field to signal document state: `comprehensive`, `adequate`, `minimal`, `outdated`, `stub`.
- Extract `audiences` where identifiable (developers, end-users, operators, ai-agents, contributors).
- Use `last_modified` from git when possible: `git log -1 --format='%as' -- <path>`
- Documentation-only repos (no code) are valid — catalog everything as conceptual docs.

## Validate

- All required fields present per schema
- Enum values match schema definitions
- Document paths exist in the repository
- Coverage flags are consistent with section contents
- `readme_count` matches actual count
